# Project Vision

The goal is to build an AI-powered adaptive learning platform that can generate personalized learning roadmaps for any topic and continuously adapt those roadmaps based on learner performance.

Unlike traditional roadmap tools, this system does not simply tell users what to learn. It tells them what to learn, in what order, down to the individual step, where to learn each piece from, and how to check whether they actually learned it — and all three of those (structure, resources, tests) are generated by Gemini specifically for that learner, not pulled from a static template.

The system continuously evaluates:

* Understanding
* Confidence
* Learning style
* Progress
* Knowledge gaps

and adapts learning recommendations accordingly.

The system should feel closer to a human tutor than a roadmap generator.

# Roadmap Generation System

Purpose:

Generate a **deep, multi-level** learning roadmap for any topic, using Gemini as the actual reasoning engine for structure — not a static per-subject template. The requirement driving this section: every meaningful learning step should appear as its own node, not just the big topics.

## Hierarchy Levels

* **Level 0 — Topic (root).** e.g. "Learn Django"
* **Level 1 — Phase.** Major stage. e.g. "Python Foundations", "Web Framework Basics", "Database Layer", "Authentication & Security", "Deployment"
* **Level 2 — Topic/Module.** e.g. inside "Database Layer": "Django Models", "Django ORM", "Migrations"
* **Level 3 — Step.** Granular and concrete. e.g. inside "Django Models": "Defining a Model Class", "Field Types Overview", "Model Meta Options", "Model Relationships (FK / M2M / O2O)", "Querysets & Managers", "Model Validation", "Custom Model Methods"
* **Level 4 — Micro-step (generated on demand).** A specific practice task for one Step. e.g. "Write a model with a ForeignKey to Author"

This is what "in-depth" means in this project: the graph a learner sees is built out of Steps, not Topics. Topics and Phases are just the navigable grouping on top of that.

## Generation Strategy — Recursive Top-Down Expansion

A single Gemini call cannot reliably produce a fully detailed, hundreds-of-nodes graph in one shot without truncation or shallow output. Instead, the roadmap is built recursively:

1. **Generate Phases.** Call Gemini with the topic + learner context (prior experience, goal, time budget). Output: ordered Phases with prerequisite edges between them.
2. **Generate Topics per Phase.** For each Phase, call Gemini again with that phase's name and context. Output: ordered Topics with internal prerequisite edges.
3. **Generate Steps per Topic.** For each Topic, call Gemini again. Output: the granular Steps — this is where the "every step" requirement actually lives. Each Step includes name, short description, difficulty, estimated time, and prerequisite step ids (which may reference steps in other topics/phases).
4. **Attach Resources per Step.** See Resource Recommendation Engine below — run in the same call or a follow-up call.
5. **Assemble Graph.** The backend stitches Phases → Topics → Steps into one DAG, including cross-topic prerequisite edges (e.g. a "Classes & Inheritance" step under Python Foundations feeding into the "Django Models" topic under Database Layer).

This recursive approach lets a single "Learn Django" request produce hundreds of granular, well-formed nodes without any individual Gemini response getting too large or shallow.

## Roadmap Generation Job (async)

Because recursive generation involves many Gemini calls, it runs as a background job rather than a single blocking request:

```json
{
  "job_id": "",
  "topic": "",
  "status": "pending | generating_phases | generating_topics | generating_steps | curating_resources | ready | failed",
  "progress_percent": 0
}
```

The frontend polls (or subscribes to) this job and renders the graph incrementally: Phases appear first, then expand into Topics, then Steps, rather than waiting for full depth to finish.

# Knowledge Graph

Roadmaps are represented as a **multi-level Directed Acyclic Graph (DAG)** where every Phase, Topic, and Step is its own node — this is what gives the graph its depth.

Library:

NetworkX

Node types:

* `phase`
* `topic`
* `step`
* `micro_step` (generated on demand)

Edge types:

* `contains` — parent → child hierarchy edge
* `prerequisite` — must complete source before target
* `suggests` — optional, non-blocking related node

Cross-cutting prerequisites are explicit edges even across different Phases — e.g. an "OOP" Step under one Phase pointing to a "Django Models" Topic under another. This is exactly the multi-parent case NetworkX/DAG was originally chosen for, now expressed at step-level granularity instead of just topic-level.

```json
{
  "nodes": [
    {
      "id": "",
      "type": "phase | topic | step | micro_step",
      "title": "",
      "description": "",
      "parent_id": null,
      "difficulty": "beginner | intermediate | advanced",
      "estimated_time_minutes": 0,
      "mastery": 0,
      "confidence": 0,
      "status": "not_started | in_progress | completed | weak",
      "resources": [],
      "has_test": true
    }
  ],
  "edges": [
    {
      "source": "",
      "target": "",
      "type": "contains | prerequisite | suggests"

  ]

```

Frontend consumes this JSON directly. React Flow renders the graph.

Because graphs are now large (potentially 100+ nodes per roadmap), the frontend renders Phases collapsed by default, expanding into Topics and then Steps on click (progressive disclosure via nested subflows), with a minimap and node search for navigation. See Roadmap Visualization below.

# Progress Tracking

Purpose:

Track learning progression — now computed at **Step** granularity, then rolled up.

Roll-up rule:

* Topic mastery = average of its Steps' mastery
* Phase mastery = average of its Topics' mastery

Metrics:

* Started Steps
* Completed Steps
* Current Step
* Progress Percentage (per Topic, per Phase, and overall)

Formula:

```
progress = completed_steps / total_steps
```

# Confidence Tracking

Purpose:

Measure certainty, not just correctness. Confidence is captured per Step, primarily through answers submitted to the Adaptive Test Engine (see below) rather than a single generic question.

Each answer contains:

```json
{ "correct": true, "confidence": 90 }
```

Classification:

* Correct + High Confidence → **Mastered**
* Correct + Low Confidence → **Uncertain**
* Wrong + High Confidence → **Misconception**
* Wrong + Low Confidence → **Learning**

Confidence becomes part of Learning State. The Misconception classification specifically is fed back into future test generation prompts, so Gemini can target that exact misunderstanding.

## Calibration (v3.0)

Per-answer confidence (above) measures certainty *during* a test, question by question. Calibration is a separate, coarser signal: how well a learner's upfront self-assessment matched reality.

At the moment a node is marked complete, the learner gives a single self-reported confidence (0–100) *before* seeing any test question (see Roadmap Visualization → "Mark as Complete" control). Once the generated test is scored, that prediction is compared against the test-measured outcome for the node (e.g. average correctness, or the dominant classification among its answers):

* High self-reported confidence + strong test result → **Well-calibrated**
* High self-reported confidence + weak test result → **Overconfident**
* Low self-reported confidence + strong test result → **Underconfident**
* Low self-reported confidence + weak test result → **Well-calibrated** (accurately anticipated the gap)

This is reported back to the learner alongside the test result (e.g. "you rated this 85% before the test; your answers landed closer to 50%") and stored per node, so a learner's calibration trend over time — consistently overconfident in one phase, say — becomes visible data, not just a one-off score.

# Resource Recommendation Engine

Purpose:

For every Step (and, as a rolled-up "best of" list, every Topic), use Gemini to recommend real, relevant learning resources — so the roadmap says not just *what* to learn but *where* to learn it.

Input:

```json
{
  "node_title": "",
  "node_description": "",
  "difficulty": "",
  "preferred_style": ""
}
```

Output:

```json
{
  "resources": [
    {
      "title": "",
      "type": "official_docs | video | article | interactive | course | book",
      "url": "",
      "estimated_time_minutes": 0,
      "why_recommended": "",
      "is_free": true
    }
  ]
}
```

Rules:

* Prefer official documentation and well-known free resources over obscure ones.
* Always include at least one resource matching the learner's `preferred_style` when known.
* Target 2–4 resources per Step (depth without overwhelm). Topics surface an aggregated "best of" list pulled from their child Steps rather than a separate generation call.

Reliability note: LLM-suggested URLs can be stale or occasionally hallucinated. Before persisting or displaying a resource, run a lightweight backend validation (HTTP HEAD/GET check); if a URL fails validation, fall back to a generic search link (e.g. a docs-site search query) rather than showing a dead link.

# AI Test / Assessment Generation Engine

Purpose:

Replace static quizzes with tests generated by Gemini, per learner, per Step, based on that learner's actual experience so far — not a fixed question bank.

Input:

```json
{
  "step_id": "",
  "step_title": "",
  "mastery": 0,
  "confidence": 0,
  "self_reported_confidence": 0,
  "preferred_style": "",
  "previous_wrong_answers": [],
  "learning_style_history": {}
}
```

`self_reported_confidence` is the prediction captured at mark-complete, before this test exists (see Roadmap Visualization → "Mark as Complete" control). A high value here can prompt Gemini to include at least one question that probes for a common misconception, rather than only confirming what the learner already believes; a low value can favor more conceptual, confidence-building questions.

Output:

```json
{
  "questions": [
    {
      "id": "",
      "type": "mcq | short_answer | code_completion | scenario",
      "prompt": "",
      "options": [],
      "correct_answer": "",
      "explanation": "",
      "difficulty": "beginner | intermediate | advanced",
      "confidence_prompt": true
    }
  ]
}
```

Adaptive behavior:

* Low prior mastery → more beginner-level, conceptual questions.
* High mastery but low confidence → questions designed to confirm and reinforce correctness (confidence-building).
* High confidence + history of wrong answers → harder questions that target the specific misconception (the Misconception classification from Confidence Tracking feeds directly into this prompt).
* A self-reported confidence well above prior mastery → at least one question deliberately targets a likely misconception rather than just confirming the self-rating.
* Question-type mix is weighted by `preferred_style`.

Each submitted answer still uses the existing `{correct, confidence}` structure, so results plug straight into the existing Confidence Tracking classification and update `test_history` in Learning State. The node's `self_reported_confidence` and the resulting Calibration label (Well-calibrated / Overconfident / Underconfident — see Confidence Tracking → Calibration) are stored alongside that same `test_history` entry.

Trigger: as of v3.0, this engine is invoked automatically the moment a learner marks a node complete (see Roadmap Visualization → "Mark as Complete" control), in addition to remaining available on demand.

# Roadmap Visualization

## Design goal: visibly better than the roadmap.sh-style flowchart

The baseline to beat: flat colored rectangles, right-angle dotted/solid connector lines, manually-placed checkpoint boxes, and unrelated UI (social-share icons, a floating "AI Tutor" bar) overlapping the graph itself. This project should look like a deliberately designed product, not a flowchart export. Concretely:

* **Layout**: auto-layout via a hierarchical graph algorithm (e.g. dagre/elk through React Flow), not manual box placement — Phases flow with consistent spacing and never-overlapping connectors.
* **Connectors**: smooth curved/bezier edges, not right-angle dotted lines. `prerequisite` edges are solid; `suggests` edges are a lighter, dashed stroke.
* **Node cards**: rounded-corner cards with a soft shadow, not solid-fill colored boxes. Status is communicated by a left-edge accent bar and a subtle icon, not by flooding the whole card with color.
* **Icons per node type**: a small icon (book = topic/step, flag = checkpoint, server/cloud = infra-related step) so node type is readable at a glance — unlike the reference image's plain black "Checkpoint" boxes.
* **No floating chrome over the canvas**: no docked chat bar, no share icons, no stray text labels sitting mid-graph. Any assistant/chat entry point lives collapsed in the floating toolbar until clicked.

## Layout style: structured hierarchical graph, org-chart style (v3.0)

This is the core fix to the reference image's layout, where boxes are placed by hand and lines zig-zag wherever they need to. The roadmap must read as a clean, leveled graph:

* **Strict leveling**: every node sits on a level determined by its type — all Phase nodes occupy level 1, all Topics belonging to an expanded Phase occupy level 2 directly beneath/beside it, all Steps belonging to an expanded Topic occupy level 3, and so on. Levels are visually aligned on the same row (top-to-bottom layout) or column (left-to-right layout) — never staggered the way the reference image's boxes are.
* **Org-chart branching**: when a node has multiple children or multiple prerequisite parents (the multi-parent case the DAG/NetworkX structure exists for), the connectors branch and merge the way an org chart's reporting lines do — a single trunk splitting cleanly into evenly-spaced children, not separate lines each finding their own path and crossing one another.
* **Minimal crossings**: the layout algorithm (dagre or elkjs layered mode, rank direction TB or LR) is run specifically in its "layered/hierarchical" mode, which optimizes node ordering within each level to minimize edge crossings — this is what makes it look like a properly drawn graph instead of a flowchart someone assembled by hand.
* **Consistent sizing per level**: all nodes at the same level share the same card size, so the rows/columns themselves look ordered, the way ranks in an org chart line up.

### Dynamic layout, not a static image

"Dynamic" applies to both the data and the drawing:

* **Dynamic data**: the graph is rebuilt as Gemini's recursive generation streams in — Phases render first, then each Phase's Topics, then each Topic's Steps — exactly as described in the Roadmap Generation System.
* **Dynamic layout**: every time the graph's node set changes — a level finishes generating, a node is expanded/collapsed, or a node is marked complete — the layout algorithm re-runs and existing nodes animate (smoothly slide/resize) into their new positions rather than the canvas jumping or redrawing instantly. Expanding a Phase doesn't just reveal its Topics; the whole graph re-balances around them.
* **Stable identity**: nodes keep their identity (id, position memory) across re-layouts where possible, so a learner's mental map of "where things are" isn't constantly disrupted — only the affected branch reflows.

## Hidden "Mark as Complete" control (v3.0)

Each node has a completion control that is **not** a big checkbox sitting permanently in the open (unlike the reference design's static checkpoint boxes). It reveals itself only on interaction:

* Clicking it doesn't close the loop instantly — it swaps the checkmark for a tiny inline prompt, right on the card: "How confident do you feel about this?" (a compact slider or Low/Medium/High quick-pick, on the same 0–100 scale used everywhere else in the app). One tap or drag, then confirm. No full modal — this stays quick and inline.
* Once that self-rating is given, the node flips to `completed` (status → Green) and the control fades back out — the canvas stays clean by default either way.

### What happens after marking complete

Marking a node complete is not just a status flip — it immediately drives the Adaptive Test Engine:

1. The self-reported confidence (0–100) just captured is stored against this node — note this is a *prediction*, taken before any test exists, not a measurement.
2. Node status → `completed`; Learning State updated.
3. Backend immediately calls `POST /ai/generate-test` for that node's `step_id`, using Gemini, scoped to exactly the topic just studied — passing the self-reported confidence alongside the usual mastery/history context (see AI Test / Assessment Generation Engine), so a surprisingly high self-rating can prompt Gemini to include a question that specifically probes for a common misconception rather than only confirming what the learner already believes.
4. The Test Modal opens automatically with that freshly generated test — the learner doesn't have to navigate anywhere to find it.
5. Submitted answers feed back into Confidence Tracking as usual (Mastered / Uncertain / Misconception / Learning), which can flip the node's status again (e.g. back to Amber if the test reveals a misconception).
6. **Calibration check**: the upfront self-reported confidence (the prediction) is compared against the test-measured outcome (the reality). See Confidence Tracking → Calibration below — this comparison is itself a signal worth surfacing, independent of mastery.

## Progressive Disclosure

Because a fully expanded deep roadmap can have 100+ nodes, collapse/expand-by-default remains a usability requirement:

* Default view: Phases only (5–8 nodes), collapsed.
* Click a Phase → expands inline into its Topics (subflow).
* Click a Topic → expands into its Steps.

## Theme: Light, by default (changed in v3.0)

Light theme is now the default and primary target, not dark mode. A light, airy canvas reads as cleaner for a graph-centric page and contrasts well against the reference design being improved on.

* Background: off-white (`#FAFAFB`), not pure white — reduces glare on a large canvas
* Cards/panels: pure white (`#FFFFFF`) with a 1px hairline border (`#E7E7EA`) and a soft, low-opacity shadow for elevation, instead of a flat color fill
* Accent: one confident accent color for primary actions (e.g. a single indigo/blue); status colors (gray/blue/green/amber) stay muted/pastel rather than fully saturated, so the canvas doesn't read as a wall of bright boxes the way the reference flowchart does
* Text: near-black (`#16161A`) for primary text, mid-gray (`#6B6B75`) for secondary — never pure `#000000` on pure white, for comfortable reading
* Typography: one geometric sans for UI (e.g. Inter), one monospace for code/JSON on the Debug page
* Spacing: 8px base scale
* Graph nodes: rounded-corner cards with soft shadow, consistent sizing per node type — Phase nodes visually larger/bolder than Step nodes; status communicated via a left-edge color bar plus a faint tint, not a fully colored fill

Dark mode becomes the optional future variant (reversed from v2.0).

Component library: shadcn/ui as the base, with custom node components for React Flow.

# Success Criteria

A learner should be able to:

1. Enter any topic.
2. Receive a deep, multi-level roadmap (Phases → Topics → every individual Step) — not just a high-level outline.
3. View the roadmap as a collapsible/expandable DAG, on a single full-bleed page with nothing else competing for attention, that stays readable and visually clean even with 100+ nodes.
4. See curated, relevant learning resources attached to every step.
5. Track progress at step-level granularity, rolled up to topics and phases.
6. Track confidence.
7. Discover their learning style.
8. Receive AI recommendations grounded in their actual mastery/confidence data.
9. Mark any step complete with a single discreet action, and immediately receive a test generated specifically for that step — not a static quiz, and not a separate manual step.
10. Request alternative explanations.
11. Submit projects.
12. Receive project feedback.
13. Learn through a fully personalized, Gemini-powered system from roadmap to resource to test, on a clean, light, single-page interface.
